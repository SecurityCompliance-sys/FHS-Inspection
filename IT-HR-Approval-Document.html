1

ภาพรวมและวัตถุประสงค์ของระบบ

FHS Inspection (Fire Hose Station Inspection) คือระบบดิจิทัลสำหรับการตรวจสอบและบริหารจัดการสถานีหัวดับเพลิงภายในคลังสินค้าและไซต์งานของ BevChain Thailand พัฒนาขึ้นเพื่อแทนที่กระบวนการตรวจสอบแบบกระดาษ ซึ่งเสี่ยงต่อการสูญหายและไม่สามารถติดตามผลได้แบบ real-time

ℹ️
ระบบนี้พัฒนาในรูปแบบ Progressive Web App (PWA) สามารถติดตั้งและใช้งานบนสมาร์ทโฟน/แท็บเล็ตได้โดยไม่ต้องผ่าน App Store ทำงานได้แม้ไม่มีสัญญาณอินเทอร์เน็ต (Offline Mode)

วัตถุประสงค์หลัก

  • บันทึกและติดตามผลการตรวจสอบสถานีหัวดับเพลิงรายสถานีอย่างเป็นระบบ
  • ลดความเสี่ยงด้านความปลอดภัยอัคคีภัยด้วยการแจ้งเตือนเมื่อพบข้อบกพร่อง
  • สร้างหลักฐานเอกสารสำหรับการตรวจสอบจากหน่วยงานภายนอก (Audit Trail)
  • บริหารจัดการกรณีอุปกรณ์สูญหายผ่านระบบสอบสวน CCTV
  • สร้างรายงานสรุปเพื่อการบริหารความเสี่ยงด้านความปลอดภัยอาชีวอนามัย

สถาปัตยกรรมระบบและกระแสข้อมูล

3.1 องค์ประกอบระบบ

ชั้นเทคโนโลยีหน้าที่
Frontend / UI HTML5, JavaScript (ES Module), PWA แสดงผลและรับ Input จากผู้ใช้
Authentication Firebase Firestore + SHA-256 Hash + Salt ยืนยันตัวตนผู้ใช้ (ไม่ใช้ Firebase Auth)
Database Google Firebase Firestore (NoSQL) จัดเก็บข้อมูลสถานี, ผู้ใช้, ผลการตรวจ
Image Storage Cloudinary CDN จัดเก็บรูปภาพหลักฐานการตรวจ
Offline Cache IndexedDB + Service Worker ทำงานได้เมื่อไม่มีอินเทอร์เน็ต
Hosting GitHub Pages / Web Server ให้บริการไฟล์ HTML/JS/CSS

3.2 กระแสข้อมูล (Data Flow)

การบันทึกผลการตรวจ
  1. ผู้ใช้สแกน QR Code / NFC บนสถานี → แอปโหลดข้อมูลสถานีจาก Firestore
  2. ผู้ใช้เข้าสู่ระบบด้วยรหัสพนักงาน + รหัสผ่าน → ตรวจสอบกับ Hash ใน Firestore
  3. ระบบดึงพิกัด GPS → เปรียบเทียบกับพิกัดสถานี → แสดงระยะห่าง
  4. ผู้ใช้กรอก Checklist + แรงดัน + สถานะอุปกรณ์
  5. ผู้ใช้ถ่ายรูป → อัปโหลดไปที่ Cloudinary → ได้รับ URL กลับมา
  6. ข้อมูลทั้งหมดถูกรวมและบันทึกลง Firestore ใน collection inspections
  7. หากพบอุปกรณ์สูญหาย → ระบบสร้าง document ใน collection cases

4

ข้อมูลส่วนบุคคลที่ระบบประมวลผล (PDPA)

⚠️
ระบบนี้เกี่ยวข้องกับ พระราชบัญญัติคุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562 (PDPA) เนื่องจากมีการเก็บรวบรวมและประมวลผลข้อมูลส่วนบุคคลของพนักงาน

4.1 ประเภทข้อมูลส่วนบุคคลที่เก็บรวบรวม

ประเภทข้อมูลตัวอย่างฐานทางกฎหมาย PDPAความจำเป็น
ข้อมูลประจำตัวพนักงาน รหัสพนักงาน, ชื่อ-นามสกุล สัญญาจ้างงาน / ประโยชน์โดยชอบด้วยกฎหมาย สูง — ระบุผู้ตรวจ
ข้อมูลการทำงาน ไซต์, บทบาท, วันเวลาเข้าใช้งาน สัญญาจ้างงาน สูง — บันทึกกิจกรรม
ข้อมูลพิกัดตำแหน่ง (GPS) ละติจูด/ลองจิจูด ณ เวลาตรวจ ประโยชน์โดยชอบด้วยกฎหมาย ปานกลาง — ยืนยันว่าตรวจในสถานที่จริง
รูปภาพ ภาพสถานีหัวดับเพลิง (อาจมีภาพบุคคลบางส่วน) ประโยชน์โดยชอบด้วยกฎหมาย ปานกลาง — หลักฐานการตรวจ
ข้อมูลความปลอดภัย จำนวนครั้ง Login ผิด, เวลาล็อค ประโยชน์โดยชอบด้วยกฎหมาย สูง — ป้องกัน Brute Force

4.2 ข้อมูลที่ระบบไม่เก็บ

  • เลขบัตรประชาชน
  • หมายเลขโทรศัพท์ส่วนตัว
  • ที่อยู่บ้าน / ข้อมูลติดต่อส่วนตัว
  • ข้อมูลสุขภาพ / ข้อมูลอ่อนไหว
  • รหัสผ่านในรูปแบบ plain text (เก็บเฉพาะ SHA-256 Hash พร้อม Salt)

4.3 วัตถุประสงค์การใช้ข้อมูล

  • ยืนยันตัวตนผู้ใช้งานระบบ
  • บันทึก Audit Trail ว่าใครตรวจอะไร เมื่อไหร่ ที่ไหน
  • สร้างรายงานสรุปผลการตรวจสอบสำหรับการบริหาร
  • ใช้เป็นหลักฐานในกรณีสอบสวนหรือ Audit

4.4 ระยะเวลาเก็บรักษาข้อมูล (Data Retention)

📌
ข้อเสนอแนะ: ระบบในปัจจุบันยังไม่มีนโยบาย Data Retention อัตโนมัติ ฝ่าย IT ควรกำหนดนโยบายดังนี้:
  • ข้อมูลผลการตรวจ (inspections): เก็บ 3 ปี เพื่อการ Audit
  • ข้อมูลเคสสอบสวน (cases): เก็บ 5 ปี
  • ข้อมูลผู้ใช้ (users): ลบออกภายใน 30 วัน หลังพนักงานออกจากงาน
  • รูปภาพ Cloudinary: เก็บ 1 ปี หรือตามนโยบายบริษัท

4.5 สิทธิ์ของเจ้าของข้อมูล (Data Subject Rights)

สิทธิ์ตาม PDPAการรองรับของระบบผู้ดำเนินการ
สิทธิ์รับทราบ (Right to be Informed)แจ้งในนโยบายความเป็นส่วนตัวของบริษัทHR
สิทธิ์เข้าถึง (Right of Access)Admin สามารถส่งออกข้อมูลในรูปแบบ ExcelIT Admin
สิทธิ์แก้ไข (Right to Rectification)Admin แก้ไขชื่อ/บทบาทผู้ใช้ได้IT Admin
สิทธิ์ลบ (Right to Erasure)Admin ลบ User ได้ (ข้อมูลการตรวจยังคงอยู่ตาม Retention)IT Admin
สิทธิ์คัดค้าน (Right to Object)พนักงานแจ้งต่อ HR เป็น Case-by-CaseHR
5

การประเมินความเสี่ยงด้านข้อมูล (Data Risk Assessment)

5.1 เมทริกซ์ความเสี่ยง

HIGH RISK
Firebase API Key ใน Source Code
API Key ถูกฝังใน HTML ที่เปิดเผยต่อสาธารณะ หากไม่ตั้ง Firestore Security Rules จะเปิดให้ใครก็ได้อ่าน/เขียนข้อมูล
HIGH RISK
Cloudinary API Key ใน Source Code
Upload Preset ถูกฝังใน HTML ทำให้บุคคลภายนอกอาจนำไปใช้อัปโหลดไฟล์เข้า account ของบริษัทได้
MEDIUM RISK
ข้อมูล Session ใน sessionStorage
ข้อมูลผู้ใช้ถูกเก็บใน sessionStorage ซึ่งโค้ด JavaScript บนหน้าเดิมอ่านได้ เสี่ยงหาก XSS โจมตีสำเร็จ (แต่มีการป้องกัน XSS แล้ว)
MEDIUM RISK
รูปภาพบน Cloudinary CDN
URL รูปภาพเมื่ออัปโหลดสำเร็จสามารถเข้าถึงได้โดยตรง หากได้รับ URL ไม่จำเป็นต้อง Login ก็ดูรูปได้
MEDIUM RISK
GPS Tracking พนักงาน
พิกัดของพนักงานถูกบันทึกทุกครั้งที่ตรวจ อาจถูกมองว่าเป็นการติดตามตำแหน่งต้องขอความยินยอม
LOW RISK
รหัสผ่านถูก Hash แล้ว
รหัสผ่านถูกเข้ารหัสด้วย SHA-256 + Salt ก่อนบันทึก ความเสี่ยงต่ำแต่ SHA-256 ไม่ใช่ Password Hashing Function ที่ออกแบบมาเฉพาะ (แนะนำใช้ bcrypt หรือ Firebase Authentication แทน)

5.2 ตารางการประเมินความเสี่ยงโดยละเอียด

ความเสี่ยงโอกาสเกิดผลกระทบระดับการจัดการ
Firestore Rules ไม่ปลอดภัย สูง (หากยังไม่กำหนด) สูงมาก Critical กำหนด Security Rules ทันที
API Key รั่วไหลสู่สาธารณะ กลาง สูง High จำกัด Domain ใน Firebase Console
Brute Force การเข้าสู่ระบบ ต่ำ สูง Medium มีระบบ Lockout 5 ครั้ง / 15 นาทีแล้ว
การเปิดเผยข้อมูลส่วนบุคคลผิดพลาด ต่ำ กลาง Medium จำกัด Role-Based Access Control
ข้อมูลสูญหายจาก Firebase ต่ำมาก สูง Medium Firebase มี SLA 99.95%, เปิด Backup
พนักงานใช้อุปกรณ์ส่วนตัว กลาง กลาง Medium Session หมดอายุเมื่อปิด Browser
6

ความเสี่ยงการรั่วไหลของข้อมูล (Data Leak Risk)

6.1 ช่องทางที่อาจเกิด Data Leak

ช่องทางที่ 1: Firebase API Key ที่เปิดเผยใน Source Code

ลักษณะความเสี่ยง: API Key ถูกฝังตรงใน HTML File ซึ่งผู้ใช้สามารถดูได้ผ่าน Developer Tools บุคคลภายนอกอาจนำ API Key ไปเขียนสคริปต์โจมตีฐานข้อมูลได้

ผลกระทบ: หากไม่มี Firestore Security Rules ที่เข้มงวด อาจทำให้ข้อมูลพนักงาน ผลการตรวจ และข้อมูลสถานีทั้งหมดถูกเข้าถึงโดยไม่ได้รับอนุญาต

มาตรการบรรเทา:

  • จำกัด API Key ให้ใช้ได้เฉพาะ Domain ที่กำหนดใน Firebase Console (HTTP Referrer Restriction)
  • กำหนด Firestore Security Rules ให้ตรวจสอบ Authentication ก่อนอนุญาตเข้าถึงข้อมูลทุกครั้ง
  • เปิดใช้งาน Firebase App Check เพื่อยืนยันว่า Request มาจากแอปที่ถูกต้องเท่านั้น
ช่องทางที่ 2: รูปภาพบน Cloudinary สาธารณะ

ลักษณะความเสี่ยง: รูปภาพที่อัปโหลดบน Cloudinary เข้าถึงได้ผ่าน URL โดยตรง (Public URL) บุคคลที่ได้รับ URL จากการรั่วไหลสามารถดูภาพได้โดยไม่ต้อง Login

ผลกระทบ: รูปภาพอาจมีภาพภายในโกดัง หรืออาจมีบุคคลปรากฏในพื้นหลัง

มาตรการบรรเทา:

  • กำหนด Folder Permissions ใน Cloudinary ให้เป็น Private
  • ใช้ Signed URL หรือ Access Control สำหรับการดูรูปภาพ
  • หลีกเลี่ยงการถ่ายรูปที่มีบุคคลปรากฏชัดเจน
ช่องทางที่ 3: Export ข้อมูลออกเป็น Excel

ลักษณะความเสี่ยง: ระบบมีฟีเจอร์ Export ข้อมูลประวัติการตรวจ, เคส, และแผนการตรวจเป็นไฟล์ Excel ผู้ที่มีสิทธิ์ Admin/Viewer สามารถนำข้อมูลออกไปได้

ผลกระทบ: ไฟล์ Excel ที่มีข้อมูลพนักงานอาจถูกส่งต่อหรือสูญหาย

มาตรการบรรเทา:

  • จำกัดสิทธิ์ Export เฉพาะ Admin เท่านั้น
  • บันทึก Log ทุกครั้งที่มีการ Export ข้อมูล
  • กำหนดนโยบาย Data Classification สำหรับไฟล์ที่ Export
ช่องทางที่ 4: อุปกรณ์สูญหาย / ถูกขโมย

ลักษณะความเสี่ยง: หากสมาร์ทโฟนของพนักงานถูกขโมยในขณะที่ยังเปิด Session อยู่

มาตรการที่มีอยู่แล้ว:

  • ใช้ sessionStorage (ล้างอัตโนมัติเมื่อปิด Browser)
  • Admin สามารถระงับบัญชีผู้ใช้ (active: false) ได้ทันที

7

มาตรการรักษาความปลอดภัย (Security Controls)

7.1 มาตรการที่มีอยู่ในระบบปัจจุบัน

  • รหัสผ่านถูก Hash ด้วย SHA-256 + Salt ก่อนจัดเก็บ (ไม่เก็บ Plain Text)
  • ระบบล็อคบัญชีอัตโนมัติหลังล็อกอินผิด 5 ครั้ง (Lock 15 นาที)
  • ป้องกัน XSS ด้วยการ Escape ทุก Input ก่อนแสดงผล
  • Role-Based Access Control (Admin / Inspector / Viewer)
  • ยืนยันพิกัด GPS เพื่อตรวจสอบว่าผู้ตรวจอยู่ในสถานที่จริง
  • Session ถูกล้างเมื่อปิด Browser (sessionStorage)
  • Admin สามารถระงับบัญชีพนักงานได้ทันที
  • Offline Mode ด้วย Service Worker — ไม่ส่งข้อมูลค้างออกไปเมื่อออนไลน์แล้ว

7.2 มาตรการที่แนะนำให้ดำเนินการเพิ่มเติม

มาตรการความสำคัญผู้รับผิดชอบกำหนดเสร็จ
กำหนด Firestore Security Rules อย่างเข้มงวด Critical IT Admin ก่อน Go-Live
จำกัด Firebase API Key ให้ใช้ได้เฉพาะ Domain ของบริษัท High IT Admin ก่อน Go-Live
เปิด Cloudinary Private Access / Signed URL Medium IT Admin ภายใน 30 วัน
เปิด Firebase App Check Medium IT Admin ภายใน 30 วัน
กำหนดนโยบาย Data Retention และตั้ง Automated Cleanup Medium IT + Legal ภายใน 60 วัน
สร้าง Privacy Notice สำหรับพนักงานที่ใช้ระบบ Medium HR + Legal ภายใน 30 วัน
เปลี่ยนมาใช้ Firebase Authentication (แทน Custom Hash) Low Developer Phase 2
เปิดใช้ Firebase Firestore Backup อัตโนมัติ Low IT Admin ภายใน 60 วัน
8

นโยบายและแนวปฏิบัติสำหรับผู้ใช้งาน

8.1 สิ่งที่ผู้ใช้ต้องปฏิบัติ

  • ใช้รหัสพนักงานและรหัสผ่านส่วนตัวเท่านั้น ห้ามใช้ Account ร่วมกัน
  • ออกจากระบบทุกครั้งหลังใช้งานเสร็จ โดยเฉพาะเมื่อใช้อุปกรณ์ส่วนกลาง
  • รายงาน IT ทันทีหากสงสัยว่าบัญชีถูก Login โดยผู้อื่น
  • ถ่ายรูปเฉพาะอุปกรณ์และสถานีหัวดับเพลิง หลีกเลี่ยงการถ่ายรูปที่มีบุคคลปรากฏ
  • ไม่แชร์ Link หรือ URL รูปภาพออกสู่ภายนอก
  • แจ้ง IT ทันทีหากอุปกรณ์ที่ใช้งานระบบสูญหายหรือถูกขโมย

8.2 สิ่งที่ผู้ใช้ห้ามทำ

  • ✕ ห้ามแชร์รหัสผ่านกับผู้อื่นไม่ว่ากรณีใด
  • ✕ ห้าม Export ข้อมูลโดยไม่ได้รับอนุญาตจาก Admin
  • ✕ ห้ามใช้ระบบเพื่อวัตถุประสงค์อื่นนอกจากการตรวจสอบสถานี
  • ✕ ห้ามพยายาม Login ด้วย Account ของผู้อื่น
  • ✕ ห้ามนำข้อมูลจากระบบไปเผยแพร่บน Social Media หรือช่องทางสาธารณะ
9

บริการภายนอก (Third-Party Services)

⚠️
ระบบใช้บริการจากผู้ให้บริการภายนอก ซึ่งมีการส่งข้อมูลออกนอกประเทศ (Cross-Border Data Transfer) กรุณาตรวจสอบให้แน่ใจว่าสัญญา Data Processing Agreement (DPA) มีผลบังคับใช้
บริการผู้ให้บริการข้อมูลที่ส่งที่ตั้งเซิร์ฟเวอร์สถานะ DPA
Firebase Firestore Google LLC ข้อมูลผู้ใช้, ผลการตรวจ, สถานี, เคส สหรัฐอเมริกา / Asia ตรวจสอบ
Cloudinary CDN Cloudinary Ltd. รูปภาพหลักฐานการตรวจ สหรัฐอเมริกา ตรวจสอบ
Google Fonts Google LLC IP Address ของผู้ใช้ (Font Loading) สหรัฐอเมริกา N/A (Technical)
html5-qrcode (unpkg CDN) npm Inc. ไม่มีข้อมูลส่วนบุคคล (Library เท่านั้น) สหรัฐอเมริกา ไม่เกี่ยวข้อง
📋
Google (Firebase) และ Cloudinary มี Standard Contractual Clauses (SCCs) และ GDPR-compliant DPA ให้บริการ แนะนำให้ Legal Team ทบทวนและลงนามก่อนเปิดใช้งาน
10

แผนรับมือเหตุการณ์ข้อมูลรั่วไหล (Incident Response Plan)

ขั้นตอนการดำเนินการผู้รับผิดชอบกรอบเวลา
1. ตรวจพบ พนักงาน / ระบบ Monitoring แจ้งเหตุต้องสงสัย พนักงาน / IT ทันที
2. ประเมิน IT ประเมินขอบเขตและประเภทข้อมูลที่รั่วไหล IT Manager ภายใน 2 ชั่วโมง
3. หยุดความเสียหาย ปิด API Key / ระงับบัญชีที่เกี่ยวข้อง / จำกัดการเข้าถึง IT Admin ภายใน 4 ชั่วโมง
4. แจ้งผู้บริหาร รายงาน MD และ Legal ทันที IT Manager ภายใน 4 ชั่วโมง
5. แจ้ง PDPA หากข้อมูลส่วนบุคคลรั่วไหล ต้องแจ้ง PDPC ภายใน 72 ชั่วโมง Legal / DPO ภายใน 72 ชั่วโมง
6. แก้ไขและป้องกัน ดำเนินมาตรการแก้ไขและปรับปรุงระบบ IT + Developer ภายใน 7 วัน
7. รายงานสรุป จัดทำรายงานเหตุการณ์และบทเรียน IT Manager ภายใน 14 วัน

สรุปและคำขออนุมัติ

11.1 สรุปความพร้อมของระบบ

หัวข้อสถานะหมายเหตุ
ฟังก์ชันการทำงานหลักพร้อมครบถ้วนตามความต้องการ
การยืนยันตัวตนพร้อมHash + Lockout มีแล้ว
Role-Based Access Controlพร้อม3 ระดับ
Firestore Security Rulesต้องดำเนินการกำหนดก่อน Go-Live
API Key Restrictionต้องดำเนินการจำกัด Domain ก่อน Go-Live
Privacy Notice สำหรับพนักงานอยู่ระหว่างดำเนินการHR จัดทำ
Data Retention Policyอยู่ระหว่างดำเนินการLegal พิจารณา

Cloudinary Private Accessอยู่ระหว่างดำเนินการภายใน 30 วัน

11.2 ประโยชน์ที่คาดได้รับ

  • ลดเวลาตรวจสอบจาก ~30 นาที/สถานี เหลือ ~10 นาที
  • สร้าง Audit Trail ดิจิทัลที่ไม่สามารถแก้ไขย้อนหลังได้
  • ตรวจจับอุปกรณ์สูญหายได้เร็วกว่าเดิมด้วยระบบ Alert
  • ลดการใช้กระดาษและการจัดเก็บเอกสาร
  • ผู้บริหารดูภาพรวมได้ Real-time ผ่าน Dashboard

11.3 เงื่อนไขก่อนนำระบบไปใช้งาน (Pre-Go-Live Checklist)

  • IT ดำเนินการตั้ง Firestore Security Rules
  • IT จำกัด Firebase API Key ให้ใช้ได้เฉพาะ Domain ของบริษัท
  • HR จัดทำ Privacy Notice และแจ้งพนักงานที่เกี่ยวข้อง
  • Legal ทบทวน DPA กับ Google (Firebase) และ Cloudinary
  • IT อบรมผู้ใช้งาน (Inspector และ Admin) ก่อนเริ่มใช้จริง
  • กำหนดผู้รับผิดชอบดูแลระบบ (System Owner) อย่างเป็นทางการ
